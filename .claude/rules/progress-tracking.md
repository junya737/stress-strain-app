# Progress Tracking — 実験進捗管理ルール

## ファイル構成

| ファイル | 役割 | 更新頻度 |
|----------|------|----------|
| `reference/progress/dashboard.md` | 現状スナップショット（ベストスコア・確定知見） | ベスト更新時・知見確定時 |
| `reference/progress/plan.md` | 次の実験計画・優先順位・中長期方針 | 計画変更時 |
| `reference/progress/experiment_log.md` | 全実験の履歴テーブル・カテゴリ別知見 | **毎実験後（必須）** |
| `reference/progress/implementation_log.md` | 実装・修正ログ（コード変更履歴・既知問題） | **実装完了・バグ発見時** |

## 実験ID採番ルール

- 形式: `E001`, `E002`, ... （3桁ゼロ埋め）
- 連番で採番。欠番は作らない
- 最新IDは `experiment_log.md` のテーブル末尾を参照

## 実装ID採番ルール

- 形式: `I001`, `I002`, ... （3桁ゼロ埋め）
- 連番で採番。欠番は作らない
- 最新IDは `implementation_log.md` の実装ログテーブル末尾を参照

## 問題ID採番ルール

- 形式: `K001`, `K002`, ... （Known issueの K、3桁ゼロ埋め）
- 連番で採番。欠番は作らない
- 最新IDは `implementation_log.md` の未修正の既知問題テーブル末尾を参照

## 更新タイミングと手順

### 実験完了時（毎回必須）

1. **experiment_log.md**: テーブルに新しい行を追加
   - 全カラムを埋める（LB未提出の場合は `—` ）
   - 結果記号（`++`/`+`/`=`/`-`/`--`）を付与
   - 関連する実装IDがあれば `関連実装` 列に記入
2. **experiment_log.md**: 新たな知見があれば「判明した知見」セクションに追記
3. **dashboard.md**: ベストスコアが更新された場合のみテーブルを更新
4. **dashboard.md**: 結論が確定した知見があれば追記

### 実装完了・コミット時

1. **implementation_log.md**: 実装ログテーブルに `I` を追加
   - バグ修正、リファクタリング、新機能追加など実験以外のコード変更が対象
   - 関連する `K`（既知問題）があれば状態を `Closed` に更新
2. **implementation_log.md**: 新たな実装知見があれば「判明した実装知見」セクションに追記

### バグ発見時

1. **implementation_log.md**: 未修正の既知問題テーブルに `K` を起票（状態: `Open`）
   - すぐに修正する場合でも、記録のために起票→修正→`Closed` のフローを推奨

### 計画変更時

1. **plan.md**: 優先順位の変更、新アイデアの追加、完了した項目の削除

### セッション開始時（推奨）

1. `dashboard.md` を読んで現状を把握
2. `plan.md` を読んで次のタスクを確認
3. `implementation_log.md` の未修正問題テーブルを確認

### セッション終了前（推奨）

1. 未修正問題テーブルが最新か確認

## experiment_log.md テーブルのカラム定義

| カラム | 説明 |
|--------|------|
| ID | 実験ID（E001〜） |
| 日付 | 実験実行日（MM/DD） |
| ベース | ベースとなったモデルまたは実験ID |
| 変更内容 | 何を変えたか（簡潔に） |
| CV geo_mean | ローカル検証のBLEU×chrF++幾何平均 |
| LB | Kaggleリーダーボードスコア |
| 結果 | `++`/`+`/`=`/`-`/`--` |
| モデルパス | 保存先パス（推論設定のみの変更なら `—`） |
| 備考 | 補足情報 |
| 関連実装 | 関連する実装ID（`I001` など。なければ `—`） |

## implementation_log.md テーブルのカラム定義

### 実装ログテーブル

| カラム | 説明 |
|--------|------|
| ID | 実装ID（I001〜） |
| 日付 | 実施日（MM/DD） |
| 種別 | `fix` / `refactor` / `feat` / `chore`（gitコミットのtypeと同じ） |
| 概要 | 何をしたか（1行） |
| 対象 | 主な変更ファイル/モジュール |
| 関連実験 | 関連する実験ID（`E025` など。なければ `—`） |
| コミット | コミットハッシュ短縮形またはブランチ名 |
| 備考 | 原因、影響、補足 |

### 未修正の既知問題テーブル

| カラム | 説明 |
|--------|------|
| ID | 問題ID（K001〜、Known issueのK） |
| 発見日 | 発見日（MM/DD） |
| 概要 | 問題の内容（1行） |
| 影響 | スコアへの影響度（`高`/`中`/`低`/`不明`） |
| 関連 | 関連する I/E ID |
| 状態 | `Open` / `Closed`（修正済みで `Closed` にしたら実装ログに I を起票） |

## 結果記号の基準

※変化量はコンペによって異なるため、あくまで目安。LBとCVの両方を考慮して総合的に判断すること。

| 記号 | LB変化 | CV変化 |
|------|--------|--------|
| `++` | +2以上 | +2以上 |
| `+` | +0.5〜2 | +0.5〜2 |
| `=` | ±0.5未満 | ±0.5未満 |
| `-` | -0.5〜-2 | -0.5〜-2 |
| `--` | -2以下 | -2以下 |
